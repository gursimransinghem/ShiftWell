# Stack Research: ShiftWell v1.1 Feature Additions

**Researched:** 2026-04-06
**Scope:** Stack additions for (1) ActivityKit Live Activities, (2) Apple Privacy Manifest, (3) HealthKit sleep additions
**Overall confidence:** HIGH — all three areas verified against npm registry, node_modules inspection, and multiple current sources

---

## Summary

This is a targeted stack delta, not a full re-research. The existing stack (Expo SDK 55, `@kingstinct/react-native-healthkit@^13.3.1`, EAS) is already correct for all three feature areas. The HealthKit work requires zero new dependencies. The Privacy Manifest requires zero new dependencies but does need explicit `app.json` config. ActivityKit is the only area requiring a new package, and it brings mandatory workflow implications that require a clear decision before implementation.

---

## Stack Additions

### 1. ActivityKit / Live Activities

**Current state:** `live-activity-service.ts` is a fully designed stub that uses notifications as fallback. The real ActivityKit path is guarded by `__LIVE_ACTIVITY_NATIVE_MODULE__`. The stub is the correct pattern — nothing needs to change until EAS Build is wired.

**The decision:** Two viable packages exist. Pick one before implementation begins.

#### Option A: `expo-live-activity` by Software Mansion (RECOMMENDED)
| Property | Value |
|----------|-------|
| Package | `expo-live-activity` |
| Latest version | `0.4.2` (April 2026) |
| Maturity | Early development, breaking changes possible in minor versions |
| Expo integration | Config plugin + `npx expo prebuild --clean` |
| Managed workflow | Does NOT require ejecting. Requires custom dev client (Expo DevClient), not Expo Go |
| Dynamic Island | Yes — via SwiftUI ActivityKit widget extension generated by the plugin |
| iOS minimum | 16.2 |

**Why:** Software Mansion is the most credible third-party Expo native module author (also makes Reanimated, Screens). Config plugin handles the widget extension target automatically. API matches the existing stub's `start / update / end` pattern. Assets go in `assets/liveActivity/`.

**Install:**
```bash
npm install expo-live-activity
```

**app.json addition:**
```json
{
  "expo": {
    "plugins": ["expo-live-activity"]
  }
}
```

**Then:** `npx expo prebuild --clean`

#### Option B: `expo-widgets` (Official Expo, alpha — NOT RECOMMENDED for v1.1)
| Property | Value |
|----------|-------|
| Package | `expo-widgets` (part of Expo SDK 55) |
| Maturity | **Alpha** — explicitly flagged by Expo as "not production ready" |
| Status | Subject to breaking changes; hasn't reached beta |
| iOS minimum | Not yet documented |

**Why not:** Expo's own documentation classifies this as alpha. It can't be recommended for a first App Store submission. Software Mansion's package is more mature.

### 2. Apple Privacy Manifest

**No new package needed.** This is configuration only.

**Background:** Since May 1, 2024, all new App Store submissions must include `PrivacyInfo.xcprivacy` declaring why the app uses restricted Apple APIs. Apple does not always aggregate dependency manifests correctly, so the app-level manifest must be explicit.

**What ShiftWell uses** (confirmed by inspecting node_modules):

| API Category | Used By | Reason Code |
|---|---|---|
| `NSPrivacyAccessedAPICategoryUserDefaults` | expo-constants, expo-notifications, expo-task-manager, react-native core | `CA92.1` (app preferences) |
| `NSPrivacyAccessedAPICategoryFileTimestamp` | expo-application, expo-file-system, async-storage, react-native core | `C617.1` (app own files), `3B52.1` (user-requested), `0A2A.1` |
| `NSPrivacyAccessedAPICategoryDiskSpace` | expo-file-system | `E174.1` (prevent write errors), `85F4.1` (display to user) |

**HealthKit** does not use required-reason APIs (no `NSPrivacyAccessedAPITypes` in its manifest). However, the app must declare collected data types in `NSPrivacyCollectedDataTypes` if health data is used. This is separate from the required-reason API list — it goes in `NSPrivacyNutritionLabel` style declarations in the App Store Connect privacy nutrition label, not in PrivacyInfo.xcprivacy.

**RevenueCat (`react-native-purchases`):** No `PrivacyInfo.xcprivacy` found in node_modules. RevenueCat's native iOS SDK (PurchasesHybridCommon) does ship one, but it arrives via CocoaPods, not npm. EAS Build pulls CocoaPods, so RevenueCat's manifest arrives automatically. However, RevenueCat collects purchase history — that must be declared in App Store Connect's privacy label, not PrivacyInfo.xcprivacy.

**`react-native-reanimated`:** No `PrivacyInfo.xcprivacy` in node_modules (v4.2.1). Known GitHub issue #5878 — uses `mach_absolute_time()` which maps to `NSPrivacyAccessedAPICategorySystemBootTime`. May generate an ITMS-91053 rejection. Add `35F9.1` defensively.

**app.json config addition** (add inside `expo.ios`):
```json
"privacyManifests": {
  "NSPrivacyAccessedAPITypes": [
    {
      "NSPrivacyAccessedAPIType": "NSPrivacyAccessedAPICategoryUserDefaults",
      "NSPrivacyAccessedAPITypeReasons": ["CA92.1"]
    },
    {
      "NSPrivacyAccessedAPIType": "NSPrivacyAccessedAPICategoryFileTimestamp",
      "NSPrivacyAccessedAPITypeReasons": ["C617.1", "3B52.1", "0A2A.1"]
    },
    {
      "NSPrivacyAccessedAPICategoryDiskSpace": "NSPrivacyAccessedAPICategoryDiskSpace",
      "NSPrivacyAccessedAPITypeReasons": ["E174.1", "85F4.1"]
    },
    {
      "NSPrivacyAccessedAPIType": "NSPrivacyAccessedAPICategorySystemBootTime",
      "NSPrivacyAccessedAPITypeReasons": ["35F9.1"]
    }
  ]
}
```

**IMPORTANT:** Fix the third entry key — it has a typo above. Correct form:
```json
{
  "NSPrivacyAccessedAPIType": "NSPrivacyAccessedAPICategoryDiskSpace",
  "NSPrivacyAccessedAPITypeReasons": ["E174.1", "85F4.1"]
}
```

### 3. HealthKit — Adaptive Brain Sleep Data

**No new dependency needed.** The project already has `@kingstinct/react-native-healthkit@^13.3.1` (latest stable is `13.4.0`). The app.json already has the config plugin wired with both `NSHealthShareUsageDescription` and `NSHealthUpdateUsageDescription`.

The existing `healthkit-service.ts` already implements:
- `queryCategorySamples` for `sleepAnalysis`
- Full sleep stage parsing: `asleepCore`, `asleepDeep`, `asleepREM`, `asleepUnspecified`, `inBed`, `awake`
- `getSleepHistory(startDate, endDate)` for multi-night range queries
- `getAverageSleepingHeartRate()` for sleeping HR
- `requestAuthorization()` for both read + write

**For the Adaptive Brain morning recalculation loop**, the existing API surface is sufficient. The loop only needs to call `getSleepHistory()` over a rolling 14-night window. No new HealthKit types are required.

**Optional upgrade to consider:**

The current version is `^13.3.1`; latest is `13.4.0`. Patch release — safe to update. No breaking changes documented.

```bash
npm install @kingstinct/react-native-healthkit@13.4.0
```

**What the adaptive brain does NOT need (keep bundle lean):**

- Heart rate variability (HRV) — explicitly deferred to v1.1+ per PROJECT.md
- Apple Watch workouts data — not needed for Phase 1
- `HKQuantityTypeIdentifier.restingHeartRate` — not needed until Apple Watch HRV phase

## Warnings

### WARN-1: Live Activities require Apple Developer enrollment — HARD BLOCKER
ActivityKit widget extensions require signing with an Apple Developer certificate. EAS Build will fail without an enrolled account. Do not install `expo-live-activity` until the Apple Developer account is active. The current stub is the correct interim solution.

### WARN-2: `expo-live-activity` is early-stage
Software Mansion labels it explicitly: "breaking changes can be introduced in minor version upgrades." Pin the version exactly (`0.4.2`, not `^0.4.2`) to avoid surprise breakage on `npm install`.

### WARN-3: No managed workflow ejection required — but Expo Go is gone
Installing either ActivityKit option disables Expo Go for development. You must use `eas build --profile development` or `expo-dev-client` locally. This is expected for an app already using HealthKit (which also cannot run in Expo Go).

### WARN-4: Privacy Manifest omission = App Store rejection
ITMS-91053 is the rejection code for missing API declarations. It is enforced since May 2024. The `privacyManifests` config must be present in app.json before submitting the first TestFlight build to App Store Connect. This is the lowest-effort, highest-impact fix in the entire v1.1 stack.

### WARN-5: RevenueCat privacy labels are App Store Connect, not PrivacyInfo
RevenueCat collects purchase history. This must be declared in the App Store Connect privacy nutrition label UI (not PrivacyInfo.xcprivacy). RevenueCat's own CocoaPods-delivered manifest handles the technical API declaration — you handle the business-level disclosure in App Store Connect.

### WARN-6: HealthKit sleep data requires Apple Watch for stage data
On iPhone-only (no Apple Watch), HealthKit only records `inBed` and `asleepUnspecified` — no stage breakdown. The adaptive brain must handle this gracefully: fall back to `totalSleepMinutes` only, skip stage-based weighting. The existing `aggregateSleepSamples()` already handles this correctly via `unspecifiedSleepMinutes`.

### WARN-7: react-native-reanimated privacy manifest may cause ITMS-91053
Version 4.2.1 does not ship a `PrivacyInfo.xcprivacy`. The known GitHub issue (#5878) is open. Including `NSPrivacyAccessedAPICategorySystemBootTime: 35F9.1` in the app-level manifest is the defensive fix. This is already included in the config above.

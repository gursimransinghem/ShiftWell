  <action>
    **30-DAY-CONVERGENCE-REPORT.md structure:**

    **Executive Summary** (3-4 sentences)
    - Did the algorithm converge?
    - What percentage of users achieved the <15 min target?
    - Key finding for go/no-go on v1.3 build.

    **Study Overview**
    - N users, date range, inclusion criteria (reference VALIDATION-PLAN.md)
    - Algorithm parameters in force: α=0.3, dead zone=20 min, cap=30 min/cycle

    **Primary Analysis: Convergence**

    Wilcoxon signed-rank test on paired observations:
    - H0: no difference in median bedtime deviation (nights 1-7 vs nights 22-28)
    - H1: median deviation lower in nights 22-28
    - Report: W statistic, p-value, effect size (r = Z/√N)
    - Interpret: p < 0.05 AND r > 0.3 = statistically and practically significant

    Convergence rate:
    - % of users with nights 22-28 avg deviation < 15 min (primary success criterion)
    - % of users with convergenceStatus='converged' at day 30
    - Median nights to first convergence event

    **Secondary Analysis: Sleep Quality**
    - Mean adherenceScore at weeks 1, 2, 3, 4 (trend)
    - Sleep debt severity distribution (none/mild/moderate/severe) at start vs end
    - Night shift vs day shift breakdown (separate convergence rates if N sufficient)

    **Outlier Analysis**
    - N users with worsened deviation — what patterns?
    - N users who stalled — algorithm intervention needed?
    - % nights where feedback paused due to protocol (expected: 15-30% for shift workers)

    **Failure Criteria Assessment** (from VALIDATION-PLAN.md)
    Evaluate each:
    - [ ] Convergence failure: < 40% reach < 15 min by night 14 → PASS/FAIL
    - [ ] Regression: Sleep debt increased > 1h average → PASS/FAIL
    - [ ] Technical: > 20% nights missing data → PASS/FAIL

    **Conclusion**
    - GO: Algorithm converges as designed, proceed to Phase 17 Growth Engine
    - NO-GO: Algorithm requires parameter tuning before v1.3 build — implement recommendations

    ---

    **ALGORITHM-TUNING-RECOMMENDATIONS.md structure:**

    Based on the convergence report findings, recommend specific parameter changes:

    **If alpha=0.3 too slow** (stalled users, slow convergence):
    - Recommendation: Increase alpha to 0.4 for users with >21-night history and no recent overrides
    - Expected impact: Faster adaptation, higher variance (more oscillation possible)
    - Risk: Noisy HealthKit data amplified — test with synthetic noisy data first

    **If dead zone=20 too conservative** (too few adjustments for most users):
    - Recommendation: Reduce to 15 min (matches Phase 13 research upper bound for Apple Watch accuracy)
    - Expected impact: More frequent small adjustments, faster convergence
    - Risk: Over-adjusting on measurement noise — monitor deviation variance

    **If cap=30 too aggressive** (oscillation observed in outliers):
    - Recommendation: Reduce to 20 min per cycle for users with high deviation variance
    - Expected impact: Smoother convergence, slower for large systematic deviations
    - Risk: Slower convergence for users with >45-min systematic deviation

    **If protocol interference is high** (>40% nights paused):
    - Recommendation: Implement "protocol-aware feedback" — feedback resumes immediately after protocol ends, using pre-protocol baseline rather than full reset
    - Rationale: Heavy shift workers never reach convergence because protocols constantly interrupt

    Each recommendation should include:
    - Which algorithm constant to change
    - From → to values
    - Affected user segment
    - Expected convergence rate change
    - Implementation file: src/lib/adaptive/feedback-engine.ts (specific constant name)
    - Priority: HIGH / MEDIUM / LOW
  </action>

    - Rosen et al. (2019) — Pediatric vs adult wearable accuracy — note: adult data most applicable.

    For each source document: authors, year, journal, sample size, device tested, accuracy metric (Cohen's kappa, epoch agreement %), key finding, and ShiftWell implication.

    **Domain 2: Apple Watch Specifically (minimum 3 sources)**

    Apple Watch uses photoplethysmography (PPG) + accelerometry. Key limitations for shift workers:
    - Daytime sleep accuracy lower than nighttime (environmental light interference)
    - Stages (N1/N2/N3/REM) available from watchOS 9+ via HKCategoryValueSleepAnalysis
    - Validation: Chinoy 2021 (Apple Watch Series 6), Natale 2021, Duking et al. 2020
    - Key finding for ShiftWell: use `asleepStart`/`asleepEnd` for feedback timing rather than staging

    **Domain 3: Feedback Loop Architectures (minimum 4 sources)**

    - Borbely (1982) — Two-Process Model. Process S (sleep pressure) accumulates during wake, dissipates during sleep. Feedback loop must calibrate Process S homeostatic time constant (tau_s) from real behavior. This is the scientific basis for the convergence algorithm.
    - Phillips et al. (2017) — Updating two-process model parameters from actigraphy data. Direct precedent for our approach.
    - Skeldon et al. (2016) — Mathematical model fitting to individual sleep data. Shows convergence in 5-7 cycles.
    - Golombek & Rosenstein (2010) — Circadian clock resetting — relevant for feedback bounds during transitions.

    **Accuracy implications for implementation** (required section):
    - Total sleep duration: ±20 min measurement error → apply 20-min dead zone before feedback triggers
    - Bedtime detection: ±10 min for Apple Watch in ideal conditions → 15-min threshold is appropriate
    - Recommendation: Use `asleepStart` not `inBedStart` for feedback (removes pre-sleep reading in bed latency)
    - Daytime sleepers (night shift): validate that asleepStart accuracy holds for 9 AM-5 PM sleep windows
    - Missing data policy: if HK data absent for >3 consecutive nights, pause feedback (don't drift on stale estimates)

    **ALGORITHM-SPEC.md — Convergence Algorithm Specification**

    Structure this document as follows:

    **1. Overview**
    The HealthKit Feedback Engine adjusts planned sleep windows based on systematic deviation between planned and actual sleep timing. It implements an exponential moving average (EMA) to smooth noisy HealthKit data, converging on the user's true chronotype within 7 nights.

    Scientific basis: Process S homeostatic calibration (Borbely 1982), EMA smoothing for noisy actigraphy (Phillips et al. 2017).

    **2. Inputs**
    - `discrepancyHistory: SleepComparison[]` — last 7 nights (from Phase 14 persistence layer)
    - `currentPlan: SleepPlan` — the plan currently in force (from plan-store)
    - `circadianProtocol: CircadianProtocol | null` — active transition protocol (from adaptive/types.ts)

    **3. Outputs**
    - `adjustedBedtimeOffsetMinutes: number` — cumulative offset from chronotype baseline (positive = later)
    - `adjustedWakeOffsetMinutes: number` — cumulative offset from algorithm wake time
    - `feedbackActive: boolean` — false when feedback is paused
    - `feedbackReason: string` — human-readable reason for adjustment or pause

    **4. Convergence Formula**

    ```
    // Exponential Moving Average of bedtime deviation
    α = 0.3  // smoothing factor (lower = more conservative, higher = faster adaptation)

    // Per-cycle update (runs once per night after HealthKit data ingested):
    if abs(actualBedtimeDeviation) < 20 min:
      // Within measurement noise floor — no adjustment (literature: ±20 min Apple Watch error)
      delta = 0

    else:
      // Smooth the signal with EMA
      smoothedDeviation = α * actualBedtimeDeviation + (1 - α) * previousSmoothedDeviation

      // Cap per-cycle adjustment at 30 min
      delta = clamp(smoothedDeviation * 0.5, -30, +30)

    // Apply to plan offset
    newBedtimeOffset = previousBedtimeOffset + delta
    ```

    Same formula applies to wake time independently.

    **5. Thresholds and Guards**

    | Condition | Behavior |
    |-----------|----------|
    | `circadianProtocol !== null` | Feedback PAUSED — protocol takes priority |
    | `abs(discrepancy) < 20 min` | No adjustment — within measurement noise |
    | `abs(delta) > 30 min` | Capped at 30 min per cycle |
    | HealthKit data missing 3+ consecutive nights | Feedback PAUSED, offset frozen |
    | `adjustedOffset` would violate minimum sleep need (≥6h) | Adjustment blocked |
    | User has overridden plan manually | Feedback PAUSED for 7 days after override |

    **6. Convergence Target**
    - Target: `avg(abs(bedtimeDeviation), last 7 nights) < 15 min`
    - Expected convergence: 5-7 nights (per Skeldon et al. 2016 for EMA with α=0.3)
    - Failure condition: after 14 nights if avg deviation still >30 min → surface alert, suggest manual review

    **7. Integration Points**
    - Called from: `src/lib/adaptive/context-builder.ts` (after Phase 15 integration)
    - Reads from: `discrepancyHistory` in plan-store (Phase 14 persistence layer)
    - Writes to: `plan-store.setBedtimeOffset(offsetMinutes)` (new action, Phase 15)
    - Blocked by: active `circadianProtocol` (checked via `adaptiveContext.protocol`)

    **8. Edge Cases**
    - First 3 nights: EMA seeded from first available deviation; no adjustment until 3 nights of data
    - Night shift users: feedback applies to daytime sleep windows; accuracy validation required (see LITERATURE-REVIEW.md Domain 2)
    - Split nights: use primary sleep block only (longest contiguous asleep segment)

    **VALIDATION-PLAN.md — 30-Day Convergence Study**

    Structure:

    **1. Study Question**
    Does the HealthKit feedback algorithm converge on the user's actual sleep timing within 7 nights? Does plan adherence improve over 30 days?

    **2. Success Metrics (primary)**
    - Convergence: Mean absolute bedtime deviation < 15 min at nights 5-7 post-activation
    - Improvement: Mean discrepancy at nights 25-30 is statistically lower than nights 1-7

    **3. Success Metrics (secondary)**
    - 70%+ of users achieve convergence target within 7 nights
    - No statistically significant increase in sleep debt during feedback period (feedback doesn't disrupt sleep)
    - Feature engagement: feedback active (not paused) for >80% of eligible nights

    **4. Sample and Duration**
    - Minimum: 20 active users with Apple Watch
    - Duration: 30 days post-activation
    - Inclusion: Wore Apple Watch on ≥5 of 7 nights per week, HealthKit authorized

    **5. Statistical Analysis**
    - Primary: Wilcoxon signed-rank test on paired (nights 1-7 vs nights 22-28) bedtime deviations
    - Use Wilcoxon (not t-test) because deviation distribution is non-normal (outlier nights, shift transitions)
    - Effect size: Cohen's d for practical significance threshold (d > 0.5 = medium, clinically meaningful)

    **6. Data Collection Points**
    - Nightly: planned bedtime, actual asleepStart (HealthKit), deviation, feedback active Y/N, current offset
    - Weekly: sleep debt severity, adherence score (from sleep-comparison.ts)
    - End of study: user-reported sleep quality (optional: PSQI-SF 4-item subset)

    **7. Failure Criteria (triggers algorithm revision before Phase 16)**
    - Convergence failure: < 40% of users reach < 15 min deviation by night 14
    - Regression: Sleep debt increases by > 1h average during feedback period
    - Technical: > 20% of nights have missing HealthKit data

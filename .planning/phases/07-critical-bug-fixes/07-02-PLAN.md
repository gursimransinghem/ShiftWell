
When `planSnapshot` is null (first run, no prior regeneration), it falls back to `currentPlan` which produces an empty delta — correct behavior for first launch.

**BUG-05 — Ensure snapshot is saved BEFORE regeneration in plan-store.ts:**

Check `setAdaptiveContext` at lines 118-127. Currently:
```typescript

  set({
```

The snapshot is already saved inside `regeneratePlan()` at lines 90-91:
```typescript
const snapshotUpdate = existingPlan
  ? { planSnapshot: existingPlan, snapshotTimestamp: new Date() }
  : {};
```

This saves the snapshot BEFORE `set({ plan, ... })` overwrites it. The snapshot flow is:
1. `setAdaptiveContext` calls `regeneratePlan()`
2. `regeneratePlan()` reads `existingPlan` from `get().plan` (the OLD plan)
3. Saves `planSnapshot: existingPlan` in the same `set()` call that updates `plan`

This is correct — no change needed in plan-store.ts. The ONLY fix needed is in useAdaptivePlan.ts to read `planSnapshot` instead of `currentPlan` for the old arg.

However, there is a timing issue: `useAdaptivePlan` computes `changes` BEFORE calling `setAdaptiveContext` (which triggers `regeneratePlan`). At that point, `planSnapshot` holds the snapshot from the PREVIOUS session's regeneration (or null on first run). This is actually correct for showing "what changed since last time" — the snapshot represents the plan before the last adaptive change.

No changes needed in plan-store.ts. Only the useAdaptivePlan.ts fix above.
